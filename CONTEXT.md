# Gym Buddy — Domain Glossary

> Voice-first AI workout companion. The app *is* your buddy — a knowledgeable friend you train with, who holds you accountable through the relationship itself (not wanting to let them down), not professional authority. Differentiation between personas is personality/character, not coaching methodology.

## Ubiquitous Language

| Term | Definition |
|------|-----------|
| **Buddy** | The AI persona the user trains with. Not a personal trainer — a knowledgeable friend who knows more than you and who you don't want to let down. Defined by the team (not user-created), ~3 at launch. Each has a distinct character, tone, voice, and system prompt, but all follow the same workout flow and use the same tools. The user picks their Buddy at onboarding. |
| **Messaging Channel** | An external chat platform (Telegram now, WhatsApp later) used for async communication outside sessions — reminders, product recommendations, check-ins. Chosen so the Buddy appears alongside the user's real contacts. |
| **Channel Conversation** | A short inbound exchange on a Messaging Channel, triggered when the user replies to a Buddy's message. Hard-capped at 5 turns. The Buddy can take lightweight actions (e.g. reschedule a workout) but must exit the conversation naturally ("I've got to run, see you tomorrow"). Not a general chat surface — no open-ended dialogue. |
| **Program** | A user's overall training plan: a named, **user-owned** collection of Workouts plus a scheduling *type* — `static` (fixed day-of-week) or `rotation` (sequence-based, advances as Workouts are completed). A user gets their first Program by adopting a Workout Template, and may then change it freely. |
| **Workout** | A named, ordered list of exercises belonging to a Program (e.g. "Push Day"). Exercises belong to the Workout, not to a schedule slot. If exercises are added or swapped, it's still the same Workout — just updated. |
| **Schedule** | Determines *when* a Workout happens. Currently day-of-week based ("Push Day on Monday"). Rotational scheduling (planned) will allow sequence-based assignment. Multiple schedule types can reference the same Workout. |
| **Session** | A single execution of a Workout by a user. Created when the user starts a workout, records start/end time, sentiment, and all logged performance. References which Workout was performed and under which Schedule. The Workout template is the plan of record — the Buddy can skip or substitute exercises within a Session, but cannot add new ones or modify the template. Deviations are logged (skipped flag, notes). |
| **Set Log** | The atomic unit of performance tracking. One row per set performed: exercise reference, set number, reps achieved, weight used, timestamp. All performance queries derive from Set Logs. |
| **Exercise Log** | A status record for an exercise within a Session: completed or skipped, plus optional notes. Does *not* carry performance data — that lives in Set Logs. (Legacy `actual_sets`/`actual_reps`/`actual_weight` columns pending removal.) |
| **Workout Template** | A team-curated **starting point** — a complete Program (Workouts + exercises + a default Schedule) such as "3-Day PPL" or "Upper/Lower 4-Day". Users adopt a Template at onboarding to avoid a blank page, then customise it without restriction (rename, add/remove/edit Workouts and exercises, change days/times). A Template is a seed, not a constraint; once adopted it becomes the user's own **Program** and the two are independent. Lives in its own team-managed catalog, separate from any user's data. |
| **Template Adoption** | The one-time **clone** of a Workout Template into a new user-owned Program at onboarding. The copy is fully independent: later edits by the user never touch the catalog, and later edits to the catalog Template never propagate to users who already adopted it. |
| **Web Client** | The browser-based surface for onboarding, payment, Workout Template selection, and account management. Exists to keep the mobile app distraction-free and to avoid app store payment fees. |
| **Mobile App** | Single-purpose native app. Its only job is running a Session — connecting to the Buddy and working out. No onboarding, no payment, no plan management. See [ADR-0001](docs/adr/0001-web-onboarding-mobile-session.md). |
| **Buddy Memory** | What the Buddy knows about a user across Sessions. Belongs to the user, not the persona — switching Buddies carries all history forward. For MVP: performance data (weights, reps, skips per exercise) and session-level sentiment/notes. Does *not* include conversation transcripts. Injected into the agent prompt at Session start. |
| **Product Recommendation** | A product link sent via the Messaging Channel. MVP is reactive only — triggered when the user mentions a need during a Session. The Buddy never proactively pushes products. Trust is the core asset; the Buddy is a friend, not a salesperson. |
| **Off-Day Session** | When a user starts a Session on an unscheduled day, the Buddy offers the *next* scheduled Workout (not a repeat of a recent one, due to muscle recovery). Allowed regardless of subscription tier, but excess usage triggers an upgrade prompt. |
| **Smart Workout Resolution** | When a user starts a Session, the system doesn't blindly follow the day-of-week schedule. It checks recent Session history and offers the next *unperformed* Workout. If the user already did Push Day on Saturday, Monday's Session becomes Pull Day instead. The underlying schedule doesn't change — Push is still "assigned" to Monday — but the Buddy adapts the current week based on what's been done. The remaining schedule slots shift accordingly for the rest of the week. |
| **Session Status** | `in_progress` → `completed` (all exercises addressed) or `abandoned` (user quit or disconnected before finishing). An abandoned Session triggers a follow-up message via the Messaging Channel — in-character for the Buddy ("You disappeared on me. Everything okay?"). |
| **Subscription Plan** | Determines how many Sessions per week the user pays for (e.g. 3 days/week). The system does not hard-block extra Sessions — the user can always train. But if they consistently exceed their plan's allowance, the Buddy (or Messaging Channel) nudges them to upgrade. Soft cap, not a hard gate. |
| **Standby Mode** | Activated by saying "mute" during a Session. The Buddy stops processing speech — only listens for the wake word (the Buddy's name, e.g. "Sergeant", "Homer"). Session clock and rest timers keep running. When the user says the wake word, the Buddy resumes and can react to elapsed time. Also available as a tap target on the mobile UI. |
| **Voice-Primary, Touch-Assist** | Sessions are voice-driven, but critical actions (log set, skip exercise, end workout, enter/exit Standby) also have tap targets on the mobile UI as fallback. The gym is hostile to voice-only interaction. |
| **Single Voice Rule** | Every user-facing message — voice during Sessions, text via Messaging Channels, push notifications — is written in the Buddy's persona. The user never sees a "system" voice. There is no generic or neutral tone; it's always their Buddy talking. |
